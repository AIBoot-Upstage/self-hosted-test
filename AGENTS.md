# Repository Working Rules

## Canonical project documents

프로젝트 작업을 시작하기 전에 다음 두 문서를 먼저 읽는다.

1. `docs/프로젝트 현황.md`: 현재 구현, 실행 흐름, 배포 상태, 한계, 다음 작업의 단일 기준
2. `docs/프로젝트 설명서.md`: 다른 사람에게 설명하기 위한 문제, 해결 방식, 핵심 개념의 기준

코드만 변경하고 문서를 나중에 맞추는 방식으로 작업하지 않는다. 아래 항목이 바뀌면 같은
commit 또는 PR에서 기준 문서를 함께 갱신한다.

- 기능 상태, API, 데이터 구조, 라우팅, workflow node 또는 실행 순서
- 외부 연동, 환경변수, 배포 방식, 운영 절차 또는 보안 경계
- 테스트 수, 검증 결과, KPI baseline, 알려진 제약 또는 다음 우선순위
- 프로젝트의 한 줄 설명, 사용자 가치, 용어 또는 외부에 설명하는 방식

## Documentation policy

프로젝트 관련 설명을 추가할 때는 새 Markdown 파일을 만드는 대신 먼저 아래 기준을 적용한다.

- 현재 사실과 진행사항은 `docs/프로젝트 현황.md`에 통합한다.
- 개념, 비개발자 설명, 발표용 핵심 메시지는 `docs/프로젝트 설명서.md`에 통합한다.
- 설치와 빠른 실행 명령은 루트 `README.md`에 유지하되 상세 설명은 기준 문서로 연결한다.
- 기존 기획서, 기술 설계서, 고도화 문서, 발표 자료는 해당 산출물 자체의 변경이 필요할 때만
  수정한다. 일상적인 진행 기록을 여러 문서에 복제하지 않는다.

새 문서는 독립된 독자와 수명 주기가 분명한 경우에만 만든다. 예시는 제출용 기획서, 발표 자료,
운영 runbook, 장애 보고서, migration guide, 외부 API 명세, 되돌리기 어려운 결정을 남기는 ADR이다.
새 문서를 만들었다면 `docs/프로젝트 현황.md`의 문서 지도에 링크하고 생성 이유를 적는다.

## Update checklist

작업을 마치기 전에 다음을 확인한다.

1. 기준 문서의 `마지막 갱신일`과 관련 상태를 수정했는가?
2. 구현 완료, 검증 완료, 목표, 미구현을 구분했는가?
3. 측정하지 않은 latency, 품질, 비용 수치를 성과처럼 기록하지 않았는가?
4. secret, token, private key, 실제 `.env` 값을 문서에 넣지 않았는가?
5. 새 문서가 꼭 필요한지, 기존 기준 문서에 통합할 수 없는지 확인했는가?
6. 수행한 테스트와 수행하지 못한 검증을 `프로젝트 현황.md`에 정확히 반영했는가?
