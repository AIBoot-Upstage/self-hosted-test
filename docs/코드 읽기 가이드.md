# 코드 읽기 가이드 (파이썬을 잘 몰라도 OK)

이 문서는 이 프로젝트의 **파이썬 코드를 어떤 순서로, 어떻게 읽으면 좋은지** 안내한다.
모든 `.py` 파일에는 한국어 주석이 달려 있으니, 이 가이드로 "지도"를 먼저 잡은 뒤
파일을 열면 훨씬 수월하다.

> 프로젝트가 무엇이고 왜 만드는지(코드 없이)는 [프로젝트 현황](프로젝트%20현황.md),
> [프로젝트 설명서](프로젝트%20설명서.md)를 먼저 보면 좋다. 이 문서는 그 다음,
> **코드 자체를 읽기 시작할 때** 보는 안내서다.

---

## 0. 목표와 마음가짐

- 목표: 이 코드를 처음부터 끝까지 다 외우는 게 아니라, **"어디에 무엇이 있는지"와
  "PR 하나가 리뷰되는 흐름"을 그릴 수 있게 되는 것**이다.
- 파이썬 문법이 낯설어도 괜찮다. 자주 나오는 문법은 [7장 미니 사전](#7-파이썬-미니-사전-이-코드에서-자주-보이는-것들)에 정리했고,
  코드 안 주석에도 처음 나올 때 설명을 붙여 두었다.
- **읽는 방법**: 파일을 열면 (1) 맨 위 모듈 docstring(파일 설명)을 읽고 → (2) 클래스/함수
  이름과 그 위 주석만 훑어 "무슨 일을 하는 함수들이 모여 있나"를 파악한 뒤 → (3) 관심 있는
  함수 하나를 골라 안쪽까지 읽는다. **처음부터 한 줄씩 읽지 말 것.**

---

## 1. 한 문장 요약

> GitHub Pull Request(PR)가 올라오면, **상황에 맞는 리뷰 전략을 자동으로 고르고**,
> **저장소 정책과 검토 절차(하네스/RAG)를 근거로** AI가 코드 리뷰를 작성해 GitHub에
> 댓글로 달아 주는 서버다.

여기서 핵심은 "아무 말이나 하는 AI"가 아니라, **근거 없는 지적을 걸러내도록 설계된
파이프라인**이라는 점이다. 그 걸러내는 장치가 바로 하네스, RAG, finding 검증이다.

---

## 2. 큰 그림: PR 하나가 리뷰되는 흐름

```
GitHub PR/체크 완료
   │  (webhook 또는 API 요청)
   ▼
[1] 특징 추출   : 이 PR의 상태를 파악 (테스트 실패? 위험 파일? 정책 있음?)
   ▼
[2] 라우팅      : 3가지 리뷰 경로 중 하나 선택 (실패진단 / 표준 / 심층)
   ▼
[3] 복잡도 분석 : (심층일 때) Radon으로 함수 복잡도 증가 측정
   ▼
[4] 하네스 선택 : diff 신호로 "검토 절차(skill)"와 "지식 카드"를 고름
   ▼
[5] 정책 검색   : (RAG) 관련 저장소 정책 chunk만 골라 옴
   ▼
[6] 프롬프트 조립: 위 재료들을 모아 모델에게 줄 지시문을 만듦 (필요 시 배치 분할)
   ▼
[7] 모델 호출   : Solar3 모델에게 리뷰를 받음 (추론 강도는 경로마다 다름)
   ▼
[8] finding 검증: 근거 없는/규칙 위반 지적을 버리고 정렬·개수 제한
   ▼
[9] 결과 조립·저장
   ▼
[10] 게시       : GitHub inline 코멘트 + 요약 댓글 (또는 로컬 파일)
```

이 흐름은 **코드에서 `review_graph.py` 한 파일에 그대로 들어 있다.** 각 단계가 "노드"라는
이름의 작은 함수 하나로 되어 있어서, 이 파일이 전체 지도 역할을 한다.

## 4. 추천 읽기 순서

아래 순서대로 읽으면 "기반 → 흐름 → 핵심 품질장치 → 바깥(입출력)" 으로 자연스럽게 넓혀 간다.
각 단계 옆의 ⏱은 대략적인 부담이다.

### 1단계 — 데이터가 어떻게 생겼는지 (⏱ 짧게, 훑기)

- **`backend/app/core/schemas.py`** ← **여기서 시작**
  프로그램이 주고받는 "데이터 상자"들의 정의다. 로직이 거의 없어 부담이 적고,
  `ReviewRequest`(입력), `ReviewRoute`(경로), `ReviewFinding`(지적), `ReviewResult`(결과)만
  눈에 익혀 두면 다음 파일들이 잘 읽힌다. 이 파일은 **주석 스타일의 기준 예시**이기도 하다.

### 2단계 — 어떤 리뷰를 할지 정하는 규칙 (⏱ 짧게)

- **`backend/app/core/routing.py`**
  PR 상태(특징)를 뽑고, 3가지 경로 중 하나를 고르는 규칙. 조건 분기가 명확해 읽기 쉽다.
  이 파일도 **주석 스타일 기준 예시**다.

### 3단계 — 전체 흐름의 뼈대 (⏱ 보통, 가장 중요)

- **`backend/app/services/review_graph.py`**
  2장의 흐름도가 코드로 된 곳. `_build_graph()`에서 노드를 연결하는 부분과,
  `_extract_features`, `_select_route`, `_retrieve_policies`, `_call_llm`,
  `_validate_findings` 같은 노드 함수들을 순서대로 보면 전체가 손에 잡힌다.
  - 파일 앞부분의 `StateGraph`/`_LocalCompiledGraph` 클래스는 **LangGraph 라이브러리가
    없을 때를 대비한 대체 구현**이다. 처음엔 건너뛰고 `ReviewWorkflowGraph`부터 봐도 된다.
- **`backend/app/services/orchestrator.py`**
  위 그래프를 실제로 조립해 실행시키는 얇은 진입점. 짧다.

### 4단계 — 리뷰 "품질"을 지키는 3대 장치 (⏱ 보통, 이 프로젝트의 핵심)

여기가 "그냥 AI 호출"과 다른 지점이다. 순서대로 읽자.

- **`review_harness/scripts/diff_signals.py`** — diff에서 신호(보안/API/성능 등)를 뽑는다.
- **`backend/app/services/policy_harness.py`** — 그 신호로 검토 절차(skill)와 지식 카드,
  정책 유형을 고른다. "무엇을 볼지"를 좁혀 오탐을 줄인다.
- **`backend/app/services/rag.py`** — 정책 문서를 검색 단위(chunk)로 쪼개고, PR과 관련 있는
  정책만 점수로 골라 온다(RAG). 여기서 검색 점수 공식을 보면 된다.
- **`backend/app/services/review_quality.py`** — 모델이 뱉은 지적을 **검증**한다. 바뀐 파일이
  맞는지, diff의 실제 줄인지, 정책/카드 근거가 진짜인지, 중복/개수 초과가 아닌지 걸러낸다.

### 5단계 — 모델에게 무엇을 어떻게 말하는지 (⏱ 보통)

- **`backend/app/services/prompt_builder.py`** — 위 재료들을 모아 모델용 지시문(프롬프트)을
  만든다. 큰 PR을 여러 "배치"로 나누는 부분도 여기 있다.
- **`backend/app/services/llm.py`** — 실제 모델 호출. `MockLLMClient`(개발/테스트용 가짜
  리뷰어)와 `LiteLLMClient`(진짜 Solar3 호출) 두 개가 있다. 먼저 Mock을 보면 이해가 쉽다.
- **`backend/app/core/config.py`** — 환경변수로 동작을 바꾸는 설정 모음. tier(low/medium/high)에
  따라 모델/추론 강도/토큰 상한이 어떻게 정해지는지 확인.

### 6단계 — 결과를 바깥으로 (⏱ 보통)

- **`backend/app/services/publisher.py`** — 리뷰 결과를 GitHub 댓글/inline 코멘트로 게시하거나
  로컬 파일로 저장한다. 마크다운을 만드는 부분(`format_review_markdown`)부터 보면 좋다.
- **`backend/app/storage/`** (`factory.py` → `local_store.py` → `postgres_store.py`) — 결과 저장.
  `factory.py`가 "무엇을 쓸지 고르는 스위치", 나머지 둘이 실제 구현이다.

### 7단계 — 서버/입구 (⏱ 보통, 필요할 때)

- **`backend/app/main.py`** — FastAPI 웹 서버. API 엔드포인트(리뷰 생성, webhook 수신,
  SSE 이벤트 스트림, 목록/조회)가 모여 있다.
- **`backend/app/services/github_app.py`** — GitHub webhook을 받아 서명 검증하고, 어떤 리뷰를
  실행할지(`review_plan`) 정하고, GitHub API로 PR/파일/체크를 수집한다. 이 프로젝트에서 가장
  긴 파일이라 **맨 마지막**에 봐도 된다.
- **`backend/app/services/events.py`** — 진행 상황을 실시간(SSE)으로 흘려보내는 이벤트 버스.
- **`backend/app/cli.py`** — 서버 없이 터미널에서 리뷰를 한 번 돌려보는 진입점. 아주 짧다.

### (선택) 평가/보조 도구

- `review_harness/scripts/evaluate_harness.py` — 하네스 선택 정확도를 수치로 측정.
- `review_harness/scripts/complexity_metrics.py` — Radon 복잡도 측정.
- `backend/app/evaluation/open_source_reviews.py`, `scripts/`, `github-action/` — 평가 데이터
  수집과 외부 연동 보조 스크립트. 처음엔 몰라도 된다.

## 5. "흐름 따라 읽기" 지름길

한 번에 하나의 함수만 따라가고 싶다면, `review_graph.py`의 `_build_graph()`를 펼쳐 놓고
노드 이름 순서대로 해당 노드 함수(`_extract_features` → `_select_route` → ...)를 읽으면 된다.
각 노드가 어느 파일의 어떤 함수를 부르는지는 아래와 같다.

| 파이프라인 노드      | 실제 일을 하는 곳                                              |
| -------------------- | -------------------------------------------------------------- |
| `extract_features`   | `core/routing.py` 의 `extract_features`                        |
| `select_route`       | `core/routing.py` 의 `select_route`                            |
| `analyze_complexity` | `review_harness/scripts/complexity_metrics.py`                 |
| `select_harness`     | `services/policy_harness.py` (+ `scripts/diff_signals.py`)     |
| `retrieve_policies`  | `services/rag.py` 의 `rank_policy_chunks` / `LocalPolicyIndex` |
| `build_prompt`       | `services/prompt_builder.py`                                   |
| `call_llm`           | `services/llm.py`                                              |
| `validate_findings`  | `services/review_quality.py`                                   |
| `persist_result`     | `storage/`                                                     |
| `publish_comment`    | `services/publisher.py`                                        |

## 6. 직접 돌려 보면서 읽기 (가장 빠른 이해법)

코드를 읽다 막히면, **실제로 한 번 실행해서 결과를 눈으로 보는 것**이 가장 빠르다.
아무 설치 없이 가짜(mock) 모델로 전체 흐름을 돌릴 수 있다.

```bash
# 가상환경 켜기
source .venv/bin/activate

# 표준 리뷰 경로: 정책을 색인한 뒤 리뷰 실행 (JSON 결과가 출력됨)
python -m backend.app.cli sample-data/review-request.json --sync-policies

# 실패 진단 경로: 테스트가 실패한 PR을 흉내낸 요청
python -m backend.app.cli sample-data/failing-review-request.json
```

출력 JSON에서 다음 키를 찾아보면, 위에서 읽은 각 단계가 실제로 무엇을 만들어 냈는지 보인다.

- `route` : 어떤 경로가 선택됐는지 ([2] 라우팅)
- `review_harness` : 감지된 신호와 고른 skill/카드 ([4] 하네스)
- `retrieved_policies` : 검색된 정책 chunk ([5] RAG)
- `finding_validation` : 지적을 몇 개 받고 몇 개를 왜 버렸는지 ([8] 검증)
- `summary`, `findings` : 최종 리뷰 요약과 지적

게시된 로컬 리뷰 댓글은 `.local-data/comments/<id>.md` 파일로도 저장된다.

하네스 선택 정확도를 수치로 보고 싶다면:

```bash
python -m review_harness.scripts.evaluate_harness
```

## 7. 파이썬 미니 사전 (이 코드에서 자주 보이는 것들)

| 문법/표현                        | 뜻 (초심자용)                                                                |
| -------------------------------- | ---------------------------------------------------------------------------- |
| `@dataclass(frozen=True)`        | 값만 담는 **불변 데이터 상자**. 만들고 나면 못 바꿔서 안전하다.              |
| `from_dict` / `to_dict`          | JSON(dict) ↔ 객체 서로 변환하는 도우미 메서드.                               |
| `@property`                      | 괄호 없이 `obj.x`로 쓰지만 실제론 계산해서 값을 주는 필드.                   |
| `@classmethod` / `@staticmethod` | 인스턴스 없이 클래스에 붙여 쓰는 함수.                                       |
| `str \| None`                    | "문자열이거나 없음(None)" 이라는 타입 표기.                                  |
| `list[X]`, `dict[str, Y]`        | X들의 목록, 문자열→Y 사전.                                                   |
| `[x for x in items if ...]`      | **리스트 컴프리헨션**: items를 훑어 조건에 맞는 걸 새 리스트로.              |
| `{x for x in items}`             | **집합** 컴프리헨션(중복 없음). `&`(교집합) `\|`(합집합) `-`(차집합).        |
| `(x for x in items)`             | **제너레이터**: 값을 하나씩 흘려보냄(주로 `sum(...)`, `any(...)` 안에서).    |
| `any(...)` / `all(...)`          | 하나라도 참 / 모두 참.                                                       |
| `with open(...) as f:`           | **컨텍스트 매니저**: 블록이 끝나면 자동으로 정리(파일 닫기, 연결 닫기 등).   |
| `try: ... except X:`             | 오류가 나도 프로그램이 죽지 않게 예외를 잡아 처리.                           |
| `replace(obj, field=값)`         | 불변 dataclass의 **일부만 바꾼 복사본**을 만든다.                            |
| `Protocol`                       | "이 메서드들만 있으면 이 타입으로 취급" 이라는 느슨한 인터페이스.            |
| `f"...{x}..."`                   | **f-string**: 문자열 안에 값을 끼워 넣기.                                    |
| `payload.get("k", 기본값)`       | dict에서 키를 꺼내되 없으면 기본값.                                          |
| `_함수이름`                      | 앞의 밑줄은 "이 파일/클래스 안에서만 쓰는 내부용"이라는 관례.                |
| `async def` / `await`            | 서버가 여러 요청을 동시에 처리하기 위한 비동기 문법(`main.py`, `events.py`). |
| `@app.get("/...")`               | FastAPI에서 특정 URL 요청을 이 함수가 처리하게 연결하는 **데코레이터**.      |

## 8. 테스트로 이해를 확인하기

각 부분이 "정확히 무엇을 보장하는지"는 테스트가 가장 잘 보여 준다. 궁금한 파일이 있으면
같은 이름의 테스트를 열어 보자. 예: `services/rag.py` → `tests/test_rag.py`.

```bash
source .venv/bin/activate
python -m pytest -q               # 전체 테스트
python -m pytest tests/test_rag.py -q   # 특정 부분만
```

테스트 함수 이름(`test_...`)이 곧 "이 코드가 지켜야 할 규칙"의 목록이다.
