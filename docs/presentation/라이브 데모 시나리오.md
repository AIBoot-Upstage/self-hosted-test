# 라이브 데모 시나리오

## 목표

품질을 하나의 절대 기준으로 제시하기 어려운 프로젝트이므로, 이 데모는 **"왜 이 리뷰를
믿을 수 있는가"**를 숫자가 아니라 **화면에서 직접 확인시키는 것**을 목표로 한다. 4분 안에
아래 세 가지를 증명한다.

1. **근거 있는 리뷰(Grounded)** — 모든 지적이 CI 로그, 저장소 정책, 공식 외부 문서 중
   하나를 실제로 인용하며, 그 출처를 클릭해서 바로 확인할 수 있다(`리뷰 근거` 절 + 지식
   카드 출처 링크).
2. **직관적인 리뷰(Intuitive)** — 리뷰 결과가 "AI가 뭘 봤는지"를 카드 ID 같은 내부
   식별자가 아니라 사람이 읽는 제목·링크·요약으로 설명한다.
3. **성장 가능성(Growth potential)** — 근거의 정확도가 감(感)이 아니라
   `evaluate_harness.py`라는 재현 가능한 지표로 관리되고, 새 검토 관점은 코드 수정 없이
   `SKILL.md`/JSON 설정 추가만으로 확장된다.

발표 중 새 PR의 전체 CI를 기다리지 않고, 같은 commit으로 미리 완료된 PR을 준비한다.

## 사전 준비

- CI 실패가 포함된 작은 PR 1개
- CI를 통과하고 **보안/로깅 관련 파일을 건드려 지식 카드가 실제로 선택되는** 작은 PR 1개
  (예: 인증 관련 파일에서 토큰을 로그로 남기는 변경 — `secret-and-sensitive-log-flow` 카드가
  선택되며 `리뷰 근거` 절에 OWASP 문서 링크 2개가 그대로 표시됨)
- 위 통과 PR에 완료된 표준 리뷰와 아직 누르지 않은 `심층 리뷰 실행` action
- Langfuse에서 해당 `review_run_id`로 검색 가능한 trace
- GitHub Actions의 최근 CD 성공 화면
- 터미널에 `python -m review_harness.scripts.evaluate_harness` 실행 결과(또는 캡처)
- 네트워크 장애 시 사용할 동일 순서의 화면 캡처 또는 3~4분 녹화본

## 시작 전 점검

```bash
curl -fsS https://<서비스-도메인>/healthz
```

```bash
sudo docker compose -p ai-code-review-agent ps
sudo docker compose -p ai-code-review-agent logs --tail=100 api
```

```bash
# 하네스 지표를 미리 한 번 돌려 화면/캡처를 준비해 둔다
python -m review_harness.scripts.evaluate_harness
```

- GitHub App이 데모 저장소에 설치되어 있는지 확인
- `CI / ci`, `AI Code Review` required check 이름 확인
- 통과 PR의 리뷰 댓글에 `### 리뷰 근거` 절과 지식 카드 출처 링크가 실제로 렌더링됐는지 확인
- 지식 카드 출처 링크(OWASP/Google 등)를 발표 전에 한 번 클릭해 실제로 열리는지 확인
- 최근 Solar3 quota와 Langfuse ingestion 상태 확인
- 브라우저 탭을 `실패 PR → 통과 PR → 지식 카드 출처 문서 → GitHub Check → Langfuse → 터미널(하네스 지표)` 순서로 열기
- token, private key, webhook secret, 내부 사용자 정보가 화면에 없는지 확인

## 4분 진행표

| 시간 | 화면 | 설명할 내용 | 증명하는 것 |
| ---: | --- | --- | --- |
| 0:00~0:25 | 실패 PR | CI 실패를 감지해 low reasoning의 실패 원인 우선 리뷰가 선택됨 | 상황별 라우팅 |
| 0:25~1:10 | 통과 PR — `리뷰 근거` 절 | 적용된 검토 절차(skill), 참조한 저장소 정책 문서, 참고한 외부 지식 카드가 한곳에 나열됨을 보여준다. 지식 카드 출처 링크 하나를 실제로 클릭해 OWASP/Google 원문 페이지를 새 탭에서 연다 | **근거 있는 리뷰** — 주장이 아니라 클릭으로 검증 가능 |
| 1:10~1:40 | 통과 PR — 개별 finding | inline finding의 `검토 기준`이 `secret-and-sensitive-log-flow` 같은 ID가 아니라 카드 제목과 출처 링크로 표시됨을 보여준다 | **직관적인 리뷰** — 내부 식별자 대신 사람이 읽는 설명 |
| 1:40~2:15 | GitHub Check action | `심층 리뷰 실행` 클릭, 사용자 선택으로만 high reasoning 실행 | 비용을 통제한 선택형 심층 분석 |
| 2:15~2:50 | 미리 완료된 심층 리뷰 | 시간·공간 복잡도, 간소화, 구조 관점이 표준 리뷰와 다른지 비교 | 독립적인 두 번째 시각 |
| 2:50~3:25 | 터미널 — `evaluate_harness.py` | skill·지식카드 recall/precision **1.0**, 정책 context **94.89%** 감소를 실시간(또는 캡처)으로 보여준다. "새 검토 관점은 `SKILL.md` 파일과 참조 JSON만 추가하면 되고, 이 스크립트가 매번 회귀를 자동 검증한다"고 설명 | **성장 가능성** — 감이 아닌 재현 가능한 지표로 관리 |
| 3:25~4:00 | Langfuse | 같은 run의 prompt, output, model, reasoning effort, token, latency 확인 | 실제 모델 호출의 투명성 |

## 데모용 변경 설계

### 실패 PR

- syntax 또는 deterministic test failure를 하나만 넣는다.
- 리뷰가 원인 파일과 실패 check를 정확히 인용하는지 보여준다.
- 보안 취약점 등 unrelated finding을 억지로 넣지 않는다.

### 정책 기반 PR (핵심 데모 PR — `리뷰 근거` 절이 가장 잘 드러나는 예제)

- **권장 예제**: 인증/로깅 관련 파일에서 토큰을 그대로 로그에 남기는 변경.
  ```python
  # 변경 후 diff 예시
  logger.info("Authorization=%s", token)
  ```
  이 변경은 `security-boundary` skill과 `secret-and-sensitive-log-flow` 지식 카드를
  선택하고, 카드 출처(OWASP Logging Cheat Sheet, Secure Code Review Cheat Sheet)가
  `리뷰 근거` 절과 finding의 `검토 기준`에 그대로 클릭 가능한 링크로 나타난다.
  로컬에서 이 결과를 미리 리허설하려면(GitHub PR 없이 mock LLM으로):
  ```bash
  python -m backend.app.cli sample-data/demo-evidence-review-request.json
  ```
  실행 후 `.local-data/comments/`에 생성된 마크다운에서 `### 리뷰 근거` 절과
  출처 링크가 그대로 나오는지 확인한다. 실제 데모에서는 이 payload와 같은 diff를
  가진 PR을 데모 저장소에 만들어 GitHub App이 리뷰하게 한다.
- finding의 `policy_source`가 실제 검색된 저장소 정책 문서(`security-and-privacy.md`)와
  일치하는지 함께 보여준다.
- 다른 예시(외부 API timeout 누락, 테스트 없는 routing 변경)로 바꿔도 되지만, 반드시
  **지식 카드 출처 링크가 실제로 걸리는 조합**을 미리 확인한 뒤 사용한다.

### 심층 리뷰 PR

- 중첩 반복문, loop 내부 I/O, 중복 branch처럼 diff만으로 입증 가능한 예제를 사용한다.
- 전체 문맥이 없으면 판단할 수 없는 복잡도 주장은 데모 finding에서 제외한다.

## 장애 시 전환 기준

| 상황 | 대응 |
| --- | --- |
| CI queue가 길어짐 | 이미 완료된 PR의 Checks 탭으로 즉시 이동 |
| webhook 지연 | GitHub delivery 화면과 서버 로그 캡처로 수신 구조 설명 |
| Upstage timeout/rate limit | 완료된 리뷰와 Langfuse trace를 사용하고 장애를 숨기지 않음 |
| Langfuse ingestion 지연 | Postgres의 provider, model, token usage 기록 캡처 사용 |
| 심층 리뷰가 발표 중 미완료 | 미리 완료된 동일 유형 PR을 새 탭에서 비교 |
| `리뷰 근거` 절에 출처 링크가 안 보임 | 사전 점검에서 확인해 둔 보안/로깅 PR로 즉시 전환(신호가 약한 diff는 지식 카드가 선택되지 않을 수 있음) |
| 출처 원문 사이트(OWASP/Google) 접속 지연 | 미리 캡처해 둔 원문 페이지 스크린샷으로 대체 |

## 발표 직후 수집할 값

- PR 생성 시각, CI 완료 시각, webhook 수신 시각, AI Check 완료 시각
- route, changed lines, changed files, input/output token, LLM latency
- batch에 선택된 정책 top-2와 finding이 인용한 정책
- 이번 데모에서 실제로 선택된 skill·지식 카드 ID와 그 출처(source_id, 문서 제목)
- `evaluate_harness.py`의 skill/지식카드 recall·precision, 정책 context 감소율
- 실패가 있었다면 단계, HTTP status, 재실행 여부
