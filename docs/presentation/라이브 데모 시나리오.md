# 라이브 데모 시나리오

## 목표

3분 안에 `CI 결과 기반 라우팅`, `정책 근거`, `선택형 심층 리뷰`, `실제 LLM 관측`을 증명한다.
발표 중 새 PR의 전체 CI를 기다리지 않고, 같은 commit으로 미리 완료된 PR을 준비한다.

## 사전 준비

- CI 실패가 포함된 작은 PR 1개
- CI를 통과하고 공통 정책 위반이 명확한 작은 PR 1개
- 위 통과 PR에 완료된 표준 리뷰와 아직 누르지 않은 `심층 리뷰 실행` action
- Langfuse에서 해당 `review_run_id`로 검색 가능한 trace
- GitHub Actions의 최근 CD 성공 화면
- 네트워크 장애 시 사용할 동일 순서의 화면 캡처 또는 2~3분 녹화본

## 시작 전 점검

```bash
curl -fsS https://<서비스-도메인>/healthz
```

```bash
sudo docker compose -p ai-code-review-agent ps
sudo docker compose -p ai-code-review-agent logs --tail=100 api
```

- GitHub App이 데모 저장소에 설치되어 있는지 확인
- `CI / ci`, `AI Code Review` required check 이름 확인
- 최근 Solar3 quota와 Langfuse ingestion 상태 확인
- 브라우저 탭을 `실패 PR → 통과 PR → Langfuse → Actions` 순서로 열기
- token, private key, webhook secret, 내부 사용자 정보가 화면에 없는지 확인

## 3분 진행표

| 시간 | 화면 | 설명할 내용 |
| ---: | --- | --- |
| 0:00~0:35 | 실패 PR | CI 실패를 감지해 low reasoning의 실패 원인 우선 리뷰가 선택됨 |
| 0:35~1:15 | 통과 PR | medium reasoning, 파일·라인 finding, 실제 정책 출처 확인 |
| 1:15~1:55 | GitHub Check action | `심층 리뷰 실행` 클릭, 사용자 선택으로만 high reasoning 실행 |
| 1:55~2:35 | 미리 완료된 심층 리뷰 | 시간·공간 복잡도, 간소화, 구조 관점이 표준 리뷰와 다른지 비교 |
| 2:35~3:00 | Langfuse | 같은 run의 prompt, output, model, reasoning effort, token, latency 확인 |

## 데모용 변경 설계

### 실패 PR

- syntax 또는 deterministic test failure를 하나만 넣는다.
- 리뷰가 원인 파일과 실패 check를 정확히 인용하는지 보여준다.
- 보안 취약점 등 unrelated finding을 억지로 넣지 않는다.

### 정책 기반 PR

- 공통 정책 문서 중 하나와 직접 대응하는 위반을 넣는다.
- 예: 외부 API timeout 누락, 민감 정보 logging, 테스트 없는 routing 변경
- finding의 `policy_source`가 실제 검색된 문서인지 확인한다.

### 심층 리뷰 PR

- 중첩 반복문, loop 내부 I/O, 중복 branch처럼 diff만으로 입증 가능한 예제를 사용한다.
- 전체 문맥이 없으면 판단할 수 없는 복잡도 주장은 데모 finding에서 제외한다.

## 장애 시 전환 기준

| 상황 | 대응 |
| --- | --- |
| CI queue가 길어짐 | 이미 완료된 PR의 Checks 탭으로 즉시 이동 |
| webhook 지연 | GitHub delivery 화면과 서버 로그 캡처로 수신 구조 설명 |
| Upstage timeout/rate limit | 완료된 리뷰와 Langfuse trace를 사용하고 장애를 숨기지 않음 |
| Langfuse ingestion 지연 | Postgres의 provider, model, token usage 기록 캡처 사용 |
| 심층 리뷰가 발표 중 미완료 | 미리 완료된 동일 유형 PR을 새 탭에서 비교 |

## 발표 직후 수집할 값

- PR 생성 시각, CI 완료 시각, webhook 수신 시각, AI Check 완료 시각
- route, changed lines, changed files, input/output token, LLM latency
- 검색된 정책 top-3와 finding이 인용한 정책
- 실패가 있었다면 단계, HTTP status, 재실행 여부
