## 발표 원칙

발표 자료의 수치는 아래 상태를 명확히 구분한다.

- **검증 완료**: 코드, 테스트, 배포 환경 또는 실제 PR에서 확인한 사실
- **목표**: 평가 전에 정의한 정량 기준
- **측정 필요**: 아직 표본 수집이나 반복 측정이 끝나지 않은 항목

측정하지 않은 latency와 리뷰 품질을 임의의 숫자로 채우지 않는다. 발표 직전에는
[성과 지표 기록표.md](./성과%20지표%20기록표.md)의 `baseline pending`만 실제 측정값으로
교체한다.

## 권장 시간 배분

| 구간 | 슬라이드 | 시간 |
| --- | ---: | ---: |
| 문제와 해결 방향 | 1~3 | 2분 |
| 동작 원리와 기술 | 4~7 | 4분 |
| 데모와 성과 | 8~9 | 2분 30초 |
| 로드맵과 결론 | 10 | 1분 |

라이브 데모가 포함되면 슬라이드 설명을 7분 내로 줄이고 데모에 2~3분을 배정한다.
