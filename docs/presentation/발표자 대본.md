# 발표자 대본

8~10분 발표 기준의 초안이다. 대괄호 항목은 발표 전에 실제 정보나 측정값으로 교체한다.

## 1. 도입

안녕하세요. `[이름]`입니다. 저희 프로젝트는 **AI Code Review Agent**입니다. 한 문장으로
설명하면, CI 결과와 저장소 정책에 맞춰 리뷰 깊이를 조절하는 GitHub App입니다. 모든 PR에
같은 비용의 AI 리뷰를 실행하지 않고, 지금 필요한 종류의 리뷰를 선택하는 데 집중했습니다.

## 2. 문제 정의

기존 코드 리뷰는 CI에서 이미 드러난 단순 실패와 설계 수준의 문제를 같은 방식으로 처리하기
쉽습니다. 또한 리뷰어마다 팀 정책을 적용하는 기준이 달라질 수 있고, 큰 diff 전체를 LLM에
전달하면 응답 시간과 비용은 늘지만 중요한 지적은 오히려 묻힐 수 있습니다. 그래서 저희는
리뷰의 성공을 댓글 생성 여부가 아니라 정확성, 근거, 시간과 비용으로 측정하고자 했습니다.

## 3. 해결 방안

GitHub App은 PR과 CI 이벤트를 받고, lint와 test 결과 및 diff를 분석합니다. 실패가 있으면
원인과 최소 수정 순서를 우선하고, 통과했다면 내부 정책 중 관련 항목을 찾아 표준 리뷰를
생성합니다. 더 넓은 관점이 필요할 때만 사용자가 GitHub Check의 심층 리뷰 버튼을 누릅니다.
하나의 Solar3 모델을 쓰되 reasoning effort를 다르게 적용하는 구조입니다.

## 4. 라우팅

자동 경로는 두 가지입니다. CI 실패는 `실패 원인 우선 리뷰`, 통과는 `정책 기반 표준 리뷰`로
갑니다. 큰 diff나 고위험 변경은 표준 결과에서 심층 검토를 권고하지만 자동으로 비싼 추론을
실행하지 않습니다. 세 번째 경로인 `선택형 심층 리뷰`는 사용자가 요청하며, 시간·공간
복잡도와 코드 간소화, 구조와 운영 영향을 독립적인 두 번째 시각에서 봅니다.

## 5. 시스템 구조

CI가 완료되면 GitHub가 webhook을 전송하고, GCP VM의 Caddy와 FastAPI를 거쳐 LangGraph가
실행됩니다. 정책은 Postgres에서 검색하고 LiteLLM을 통해 Upstage Solar3를 호출합니다.
결과는 DB에 저장한 뒤 GitHub Check와 댓글로 게시합니다. 모델 호출은 Langfuse에서 추적합니다.
배포는 GitHub Actions가 image를 build하고 WIF와 IAP로 VM에 전송해 실행 image를 갱신합니다.

## 6. LangGraph와 RAG

LangGraph는 특징 추출, 경로 선택, 하네스 선택, 정책 검색, prompt 생성, 모델 호출, 검증, 저장과 게시를
노드로 나눕니다. 현재 여섯 개 공통 정책을 heading 단위로 나누어 저장하고 신뢰된 script가
관련 skill과 정책 유형을 고른 뒤 batch별 top-2를 prompt에 반영합니다. 다만 현재 검색은 lexical overlap이며 embedding 기반 검색과 설치 저장소별 정책
자동 수집은 아직 구현 전입니다. 이 한계를 숨기지 않고 retrieval 지표로 다음 개선을 판단합니다.

## 7. 기술 선택

GitHub App과 Checks API로 사용자가 PR 화면을 벗어나지 않게 했습니다. LangGraph는 조건부
흐름을 명시하기 위해, LiteLLM은 모델 호출과 관측 인터페이스를 통일하기 위해 사용했습니다.
Postgres는 리뷰와 정책을 영속화하며 pgvector 전환 경로를 제공합니다. Langfuse는 실제 prompt,
응답, token과 latency를 보여주고 Docker image archive는 같은 실행 환경을 재현합니다.

## 8. 데모

먼저 CI 실패 PR에서 실패 원인 우선 리뷰를 확인하겠습니다. 다음은 CI 통과 PR입니다. 검색된
변경 요약, 파일별 변경 표, 정책 출처가 있는 파일·라인별 리뷰가 표시됩니다. 여기서 심층 리뷰 버튼을 누르면 복잡도와 구조를
다른 관점에서 분석한 두 번째 결과가 생성됩니다. 마지막으로 Langfuse trace에서 실제 Solar3
호출과 route, token, latency를 확인할 수 있습니다.

## 9. 성과와 평가

현재 GitHub App의 자동 리뷰, 선택형 심층 리뷰, 실제 Solar3 호출, Postgres 저장, Langfuse 추적,
GCP 자동 배포와 HTTPS까지 동작을 확인했고 단위 테스트 50개와 Ruff가 통과합니다. 모델 finding은
changed file, diff line과 정책 출처를 검증한 뒤 inline review로 게시합니다. 성능과 품질 수치는
아직 목표와 실측을 분리해 관리하고 있습니다. 하네스 선택 fixture 11개에서는 skill·지식 카드·정책
유형 Recall 1.0과 지식 카드·정책 유형 Precision 1.0, 기존 top-3 대비 정책 context 38.43% 감소를
확인했습니다. `[여기에 발표 전 측정한 p95, Recall@2,
Precision@5와 비용 절감률을 한 문장으로 추가]` 평가에는 내부 fixture뿐 아니라 대형 오픈소스의
실제 changes-requested와 approved snapshot을 활용합니다.

## 10. 향후 계획과 결론

다음 단계는 단계별 latency 자동 집계, 재시도와 중복 방지, 저장소별 정책 수집, hybrid retrieval,
큰 diff의 context selection입니다. 이를 통해 개발자는 반복 오류보다 중요한 판단에 집중하고,
조직은 리뷰 품질과 비용을 정량적으로 관리할 수 있습니다. 저희가 만들고자 하는 것은 더 많은
문장을 생성하는 리뷰어가 아니라, 필요한 순간에 근거 있는 리뷰를 제공하는 에이전트입니다.
