# 코드 리뷰 서비스 벤치마크 및 적용 결과

- 조사일: 2026-07-13
- 대상: CodeRabbit, GitHub Copilot code review, `openai/codex` 공개 PR review
- 목적: 기능 수가 아니라 개발자가 실제로 수정 판단에 사용할 수 있는 리뷰 계약과 pipeline 정의

## 1. 결론

시중 서비스와 대형 저장소의 공통점은 “더 긴 리뷰”가 아니라 다음 다섯 가지다.

1. diff 외에 저장소 지침, 주변 코드, CI와 issue 의도를 함께 사용한다.
2. PR 전체 요약과 line별 actionable finding의 역할을 분리한다.
3. correctness, security, data integrity, reliability를 style보다 우선한다.
4. finding은 발생 조건, 결과, 근거, 최소 수정안과 검증 방법을 포함한다.
5. 새 push에서는 이미 검토한 내용을 반복하지 않고 새로운 변경과 미해결 지적에 집중한다.

이 프로젝트는 CI 기반 라우팅과 선택형 추론이라는 차별점이 이미 있다. 여기에 post-generation
품질 검증, inline review, 저장소별 지침, incremental review, feedback 평가를 순서대로 결합해야
실제 효용을 인정받을 수 있다.

## 2. 제품 벤치마크

### CodeRabbit

공식 문서에서 확인한 주요 구조는 다음과 같다.

- 새 PR은 전체 리뷰, 후속 commit은 incremental review
- AI와 정적 분석 도구를 결합한 다층 분석
- category와 Critical~Info severity로 우선순위 표시
- line comment, PR summary, one-click fix와 대화형 후속 처리
- repository guideline, 과거 PR, issue, 학습된 선호, linked repository, MCP를 context로 사용
- 경로별 지침과 generated/lock/binary 파일 filter

적용 시사점은 “지식베이스가 많다”가 아니라, context의 출처와 적용 scope를 분리하고 사용자가
어떤 context가 사용됐는지 확인할 수 있게 한다는 점이다.

### GitHub Copilot code review

공식 문서에서 확인한 주요 구조는 다음과 같다.

- 자동 또는 수동 review와 새 push 재검토 설정
- Low/Medium effort를 코드 중요도와 비용에 맞춰 선택
- Actions 기반 agentic context gathering으로 repository 전체 문맥 수집
- repository-wide 및 path-specific custom instruction
- inline suggestion과 cloud agent 수정 위임
- comment별 사용자 feedback
- 사람이 결과를 검증해야 한다는 명시적 한계 표시

현재 프로젝트의 low/medium/선택형 high 라우팅은 effort routing 측면에서 방향이 맞다. 부족한
부분은 repository context와 지침의 base branch 수집, feedback 저장, inline 적용 가능 제안이다.

## 3. openai/codex PR 표본 분석

### 수집 방법

GitHub REST API의 공개 `pulls/comments`, `pulls/{number}`, `pulls/{number}/reviews` endpoint를
사용했다. comment 원문은 분석 중 임시 파일에만 저장했고 저장소에는 commit하지 않았다.

- 기간: 2026-07-06 23:13 UTC ~ 2026-07-10 20:58 UTC
- 최신 inline review comment: 1,000건
- 포함 PR: 185개
- 사람 comment: 527건
- bot comment: 473건
- 최초 comment: 686건
- reply: 314건
- line에 연결된 comment: 1,000건

bot severity 표시는 P0 11건, P1 51건, P2 370건, P3 40건, 미표시 1건이었다. 대부분이 P2인
것은 모든 지적을 merge blocker로 만들기보다 우선순위를 세분화한다는 신호다.

사람이 작성한 최초 comment 214건의 단순 keyword 분석 결과는 다음과 같다. 항목은 서로 겹치며
정답 label이 아니라 리뷰 주제 탐색용 수치다.

| 신호 | comment 수 |
| --- | ---: |
| 질문 형태 | 123 |
| 코드 제거·간소화·nit | 49 |
| correctness·failure | 38 |
| test·coverage | 33 |
| security·permission·credential | 31 |
| data·state·migration | 20 |
| performance·buffer·cache | 12 |

사람 reply 313건 중 `addressed`, `fixed`, `done`, `updated`, `added` 같은 수정 완료 신호가 164건,
`intentional`, `keep`, `already`, `not needed` 같은 반박·설명 신호가 48건이었다. AI finding의
효용은 comment 생성 수보다 **후속 수정으로 이어진 비율과 반박률**로 평가할 필요가 있다.

comment가 많은 PR 15개를 상세 조회한 표본에는 review 125건이 있었고 `COMMENTED` 119건,
`APPROVED` 6건, `CHANGES_REQUESTED` 0건이었다. 이 저장소에서는 review state만으로 좋은 코드와
나쁜 코드를 label할 수 없다. inline comment, 작성자 reply, 후속 commit을 함께 연결해야 한다.

### 표본의 한계

- 최신 나흘간의 활동이므로 장기·언어별 대표 표본은 아니다.
- keyword count는 사람이 확정한 defect category가 아니다.
- reply의 수정·반박 신호도 자동 추정이며 실제 patch 확인이 필요하다.
- 공개 데이터의 merge 여부는 품질 외 일정, 방향 변경, 중복의 영향을 받는다.

## 4. 적용한 리뷰 계약

모델 prompt와 결과 검증은 다음 계약을 사용한다.

### PR-level summary

- 리뷰 route와 선택 이유
- CI 상태와 전체 위험
- 검토 범위와 생략 범위
- inline finding 수
- 심층 리뷰 선택 가능 여부

### Line-level finding

각 finding은 다음 정보를 가져야 한다.

```text
severity / category
변경된 file과 right-side diff line
발생 조건 또는 입력
관측 가능한 영향
구체적인 diff 근거
가장 작은 수정안
집중된 검증 또는 regression test
관련 정책 출처(있는 경우)
```

칭찬, PR 변경 요약, CI 로그 반복, 취향 중심 style, 근거 없는 질문은 finding으로 게시하지 않는다.
확신할 근거가 없으면 comment 수를 채우지 않고 생략한다.

## 5. 적용한 pipeline

```text
CI completed
  -> GitHub PR/files/checks 수집
  -> feature extraction과 route 선택
  -> 관련 정책 검색 또는 skip
  -> risk-aware prompt scope 구성
  -> Solar3 review 생성
  -> finding quality gate
     -> changed file 검증
     -> right-side diff line 검증
     -> policy citation 검증
     -> 중복 제거
     -> severity 정규화와 route별 개수 제한
  -> Postgres 저장
  -> 검증된 항목은 GitHub inline review
  -> PR-level summary와 fallback comment
  -> Check 완료
```

### 이번에 코드로 반영한 항목

- actionable finding 중심의 짧은 prompt 계약
- `validate_findings` LangGraph node
- 존재하지 않는 파일 finding 제거
- diff에 없는 line을 inline 대상에서 제외
- retrieved set 밖의 정책 인용 제거
- 중복 제거, severity 정규화, route별 3/6/8개 제한
- 품질 gate의 수신·제거 사유별 건수를 review 결과와 DB payload에 저장
- 검증된 finding의 GitHub pull request inline review 게시
- inline API 실패 시 기존 PR comment fallback
- 공개 GitHub PR/review/comment 수집·정규화 CLI

### 다음 적용 순서

1. base branch의 `AGENTS.md`, `.github/copilot-instructions.md`, path instruction 자동 수집
2. 이전 reviewed head와 현재 head 사이 incremental diff
3. 주변 symbol, caller, test를 제한적으로 수집하는 repository context stage
4. comment reaction, reply, resolved 상태를 review outcome으로 저장
5. 반박·dismiss된 finding을 자동 학습하지 않고 승인 후 policy learning 후보로 관리
6. 정확한 replacement가 가능한 finding만 GitHub suggestion block 제공

## 6. 리뷰 지식 출처와 정제 계약

하네스 지식은 모델이 그럴듯하게 만든 내부 규칙이 아니라 다음 조건을 만족하는 자료에서만 만든다.

1. 표준 기관, 제품 제공자, 보안 재단 또는 대규모 엔지니어링 조직이 직접 공개한 1차 자료다.
2. PR diff에서 확인할 입력·상태·sink·실패 결과로 바꿀 수 있다.
3. 반례와 적용 제외 조건을 정의할 수 있어 취향성 지적을 줄일 수 있다.
4. URL, 제목, 기관과 수집일을 기록해 변경·폐기를 추적할 수 있다.

현재 registry는 Google Engineering Practices와 Testing Blog, OWASP Cheat Sheet, GitHub Docs,
Google API Improvement Proposals와 SRE Book, PostgreSQL 공식 문서, W3C WAI, OpenAI, Radon의
공식 자료 24개다. 원문 전체를 prompt에 넣지 않고 다음 구조의 지식 카드 27개로 정제한다.

```text
source_id와 공식 URL
  -> 적용 signal·path·patch marker
  -> 확인할 구체적 동작
  -> finding에 필요한 증거
  -> 오탐 방지 조건
  -> severity 상한
```

모든 skill과 카드는 하나 이상의 `source_id`를 가져야 한다. registry에 없는 source, 중복 ID,
HTTPS가 아닌 URL, 비어 있는 증거·오탐 필드는 하네스 초기화에서 거부한다. 외부 카드는 저장소
정책이 아니라 검토 관점이므로 `policy_source`로 출력할 수 없으며, 실제 diff 증거가 있을 때만
finding을 만드는 데 사용한다. 전체 source metadata와 카드 계약은 각각
`review_harness/references/sources.json`, `review_harness/references/knowledge-cards.json`이 기준이다.

대표 출처 그룹은 다음과 같다.

- 일반 리뷰: [Google code review](https://google.github.io/eng-practices/review/reviewer/looking-for.html)
- 테스트: [Google code coverage practices](https://testing.googleblog.com/2020/08/code-coverage-best-practices.html)
- 보안: [OWASP Secure Code Review](https://cheatsheetseries.owasp.org/cheatsheets/Secure_Code_Review_Cheat_Sheet.html)
- API·webhook: [GitHub breaking changes](https://docs.github.com/en/rest/about-the-rest-api/breaking-changes), [webhook validation](https://docs.github.com/en/webhooks/using-webhooks/validating-webhook-deliveries)
- 신뢰성: [Google SRE cascading failures](https://sre.google/sre-book/addressing-cascading-failures/)
- 데이터·동시성: [PostgreSQL explicit locking](https://www.postgresql.org/docs/current/explicit-locking.html)
- 접근성: [W3C accessibility principles](https://www.w3.org/WAI/fundamentals/accessibility-principles/)
- 하네스 설계: [OpenAI Harness Engineering](https://openai.com/index/harness-engineering/)

## 7. 평가 데이터 사용법

공개 PR 데이터는 모델 fine-tuning보다 evaluation dataset으로 먼저 사용한다.

```bash
GITHUB_TOKEN=... python -m scripts.collect_open_source_reviews \
  openai/codex --max-prs 25 \
  --output .local-data/evaluation/openai-codex.jsonl
```

수집 결과에는 PR 상태, base/head SHA, review state와 commit, inline comment, file/line, maintainer와
bot 구분이 포함된다. 원문은 `.local-data/`에만 저장하고 Git에는 올리지 않는다.

평가 label은 다음 순서로 만든다.

1. maintainer root inline comment를 finding 후보로 추출한다.
2. comment 시점 `commit_id`의 diff를 입력 snapshot으로 복원한다.
3. reply와 후속 patch로 addressed, rejected, clarification을 판정한다.
4. 모델에는 maintainer comment, reply와 이후 patch를 보여주지 않는다.
5. file, line, category, 핵심 failure scenario로 Precision@5와 Recall@5를 계산한다.
6. 정책 미주입/주입 결과를 같은 snapshot과 model setting에서 비교한다.
