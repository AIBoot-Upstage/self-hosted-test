# 현재 구조 점검 및 정량 평가 계획

작성 기준일: 2026-07-12

## 1. 결론

현재 서비스는 GitHub App, CI 이후 webhook, 규칙 기반 라우팅, LangGraph workflow,
LiteLLM/Solar3, 공통 정책 RAG, GitHub Check/댓글 게시까지 실제 동작한다. 다만 발표에서
"품질과 성능이 검증됐다"고 말하려면 다음 네 가지를 구분해야 한다.

1. 자동 리뷰는 실패 PR의 low와 정책 기반 medium이며, high는 사용자가 요청하는 두 번째 관점이다.
2. 현재 RAG는 Postgres에 chunk를 저장하지만 검색은 embedding이 아니라 lexical overlap이다.
3. 서비스의 `policies/`를 공통 정책으로 쓰며, 설치 대상 repository 정책 자동 수집은 아직 없다.
4. 평균 latency와 주관적 유용성 대신 diff 구간별 percentile과 정답 fixture 기반 품질 지표가 필요하다.

## 2. 현재 실행 순서

```text
PR opened/synchronize
  -> GitHub Actions CI
     -> lint_smoke     ┐
     -> test           ┴ 병렬 완료
  -> check_suite.completed webhook
  -> installation token / PR / files / check runs 조회
  -> AI Code Review pending Check 생성
  -> LangGraph
     -> feature extraction
     -> low 또는 medium 자동 route
     -> 관련 정책 top-3 검색 또는 skip
     -> prompt 생성
     -> LiteLLM -> Solar3
     -> Postgres 결과 저장
     -> GitHub comment + Check 완료
  -> 필요 시 사용자가 `심층 리뷰 실행`
     -> high reasoning의 독립적인 두 번째 리뷰
```

`after_checks` 모드에서는 PR 생성 시점부터 CI 완료까지의 시간이 반드시 포함된다. 따라서
서비스 latency는 CI와 리뷰 agent 시간을 분리해 보고해야 한다.

Branch protection의 required check는 병렬 job 하나가 아니라 두 결과를 합치는 `CI / ci`와
`AI Code Review`를 사용한다. 그래야 test만 먼저 통과한 상태에서 merge되는 것을 막을 수 있다.

## 3. 라우팅 점검

| 조건 | 실제 자동 동작 | 설명 |
| --- | --- | --- |
| syntax/lint/test 중 하나 실패 | `simple_failure_review`, low | 실패 근거와 최소 수정 우선순위 |
| 실패 check 없음 | `policy_context_review`, medium | 관련 정책이 검색되면 정책 기반, 없으면 일반 품질 리뷰 |
| 고위험 경로 또는 큰 diff | medium + 심층 리뷰 권고 | high를 자동 소비하지 않음 |
| 사용자가 Check action 클릭 | `deep_quality_review`, high | 다른 시각의 선택적 리뷰 |

문서에 남아 있던 "고위험 또는 대규모 diff는 자동 high" 설명은 현재 제품 방향과 다르다.
고위험 신호는 medium 결과에 심층 리뷰 필요성을 표시하고, 최종 실행 여부는 사용자가 결정한다.

## 4. 심층 리뷰 품질 기준

high 리뷰는 medium 결과를 길게 반복하면 가치가 없다. 다음 항목 중 diff로 입증 가능한 내용만
finding으로 생성한다.

1. 변경 실행 경로의 시간 복잡도와 입력 변수 기준 Big-O
2. collection, cache, buffer, recursion에 따른 공간 복잡도
3. loop 내부 network/database 호출과 반복 scan
4. 중복 branch, 불필요한 상태와 wrapper를 제거하는 동작 보존형 간소화
5. 책임 경계, 실패 격리, 보안, 유지보수성과 운영 영향

복잡도를 판단할 전체 함수나 호출 문맥이 없으면 finding을 만들지 않는다. 단순히 코드를 짧게
쓰는 취향은 간소화로 평가하지 않는다.

## 5. RAG 현재 상태와 개선 기준

### 현재 구현

| 항목 | 상태 |
| --- | --- |
| 정책 원본 | 배포 artifact의 `policies/*.md` |
| chunk | Markdown heading 기준, 최대 1,800자 |
| 저장 | Postgres `policy_chunks` |
| 검색 | PR title, file path, check summary, patch token overlap |
| 반환 | 관련 score가 있는 top-3 |
| prompt 반영 | source, section, content, score, canonical policy source |
| finding 표시 | 모델이 반환한 `policy_source`를 GitHub 댓글에 표시 |
| embedding | schema 컬럼만 존재하고 값 생성/검색은 미구현 |
| repository 격리 | 미구현. 현재 정책은 모든 설치 repository에 공통 적용 |

### 5일 내 우선순위

1. 배포마다 정책 sync를 수행하고 indexed document/chunk 수를 로그로 남긴다.
2. 정책별 positive/negative query fixture를 최소 30개 만든다.
3. retrieval `Recall@3`, `MRR@3`, 무관 정책 반환율을 측정한다.
4. 모델이 인용한 source가 실제 retrieved set에 있는지 자동 검증한다.
5. 시간이 남으면 repository ID와 source SHA를 schema에 추가하고 대상 저장소 정책 sync를 구현한다.

Embedding을 붙이기 전에는 발표에서 "pgvector semantic search"라고 표현하지 않는다. 현재 상태는
"Postgres-backed policy index와 lexical retrieval, vector 전환 가능한 schema"가 정확하다.

## 6. 리뷰 품질 평가 데이터셋

### 6.1 내부 통제 fixture

최소 36개 PR payload를 만든다.

| 유형 | 개수 | 정답 label |
| --- | ---: | --- |
| CI 실패 | 8 | 실패 check, 원인 파일, expected low route |
| 정책 위반 | 12 | policy ID, 위반 line, expected category |
| 복잡도/간소화 | 8 | Big-O 변화 또는 중복 구조, expected high finding |
| 정상 변경 | 8 | severe finding 없음, false-positive 측정 |

정답은 `expected_route`, `expected_policy_sources`, `expected_findings`, `forbidden_findings`로
저장한다. 이 세트는 prompt 변경 시마다 CI에서 재실행한다.

### 6.2 대형 오픈소스 PR 평가

단순 `merged=좋음`, `closed=나쁨` label은 사용하지 않는다. closed PR에는 중복, 방향 변경,
작성자 포기 같은 사유가 섞이고, merged PR에도 알려진 기술 부채가 있을 수 있기 때문이다.

다음과 같이 유지보수자의 실제 review state와 comment를 사용한다.

1. `python/cpython`, `django/django`, `kubernetes/kubernetes`, `rust-lang/rust`에서 표본을 수집한다.
2. `CHANGES_REQUESTED` 시점 commit을 "수정 전" 입력으로 저장한다.
3. maintainer inline comment를 file, line, category, severity 정답으로 정규화한다.
4. 같은 PR이 수정 후 `APPROVED` 또는 merge된 commit을 "수정 후" 입력으로 저장한다.
5. duplicate, stale, superseded, 문서 전용, CI 인프라 장애로 닫힌 PR은 제외한다.
6. 모델 입력에는 maintainer comment, review state, 이후 patch를 넣지 않아 label leakage를 막는다.

초기 표본은 수정 전/후 25쌍과 승인 상태 정상 PR 25건, 총 75 snapshot을 목표로 한다.
SWE-bench Verified에서 20건을 별도 추출해 실제 bug patch와 test failure에 대한 기능 결함
탐지 보조 세트로 사용한다. SWE-bench는 코드 리뷰 전용 정답은 아니므로 주 평가 세트와 섞지 않는다.

평가 근거:

- [Google Engineering Practices: What to look for in a code review](https://google.github.io/eng-practices/review/reviewer/looking-for.html)
- [GitHub REST API: Pull request reviews and comments](https://docs.github.com/en/rest/pulls)
- [SWE-bench dataset guide](https://www.swebench.com/SWE-bench/guides/datasets/)
- [SWE-bench paper](https://arxiv.org/abs/2310.06770)

## 7. 정량 KPI

### 7.1 운영 신뢰성

| 지표 | 계산식과 분모 | 초기 목표 |
| --- | --- | ---: |
| 유효 webhook 수락률 | 서명과 필수 필드가 유효한 요청 중 2xx 비율 | >= 99.5% |
| 자동 리뷰 완료율 | eligible head SHA 중 Check와 comment가 완료된 비율 | >= 98.0% |
| 일시 오류 복구율 | timeout/rate-limit 실패 중 10분 내 재시도로 완료된 비율 | >= 95.0% |
| 중복 리뷰율 | 같은 repo/PR/head/mode에 둘 이상 생성된 비율 | <= 0.5% |
| 결과 schema 유효율 | LLM 응답 중 parsing과 필수 필드 검증을 통과한 비율 | >= 99.0% |

Draft, 사용자가 취소한 실행, 새 commit으로 대체된 이전 head SHA는 완료율 분모에서 제외하되
별도 `canceled` 상태로 기록한다. GitHub/Upstage 장애는 서비스 의존성 실패로 분류하며 임의로
분모에서 빼지 않는다. 100% 성공을 주장하지 않고 실패율과 복구율을 함께 제시한다.

### 7.2 Latency

리뷰 agent latency는 `check_suite.completed webhook 수신`부터 `AI Code Review Check 완료`까지다.
PR 생성부터 CI 완료까지는 `ci_wait_ms`로 분리한다.

| Diff bucket | 기준 | p50 목표 | p95 목표 |
| --- | --- | ---: | ---: |
| Small | <= 200 lines, <= 5 files | <= 30초 | <= 75초 |
| Medium | <= 800 lines, <= 20 files | <= 60초 | <= 150초 |
| Large | <= 2,000 lines, <= 40 files | <= 120초 | <= 300초 |

두 기준 중 더 큰 bucket을 사용한다. 2,000 lines 또는 40 files를 넘으면 SLA에 억지로 포함하지
않고 `oversized`로 기록하며, 포함 파일 수와 truncation 여부를 결과에 표시한다. 평균 하나로
전체 diff를 뭉뚱그리지 않는다.

추가로 `github_fetch_ms`, `rag_ms`, `llm_ms`, `publish_ms`의 p50/p95를 route별로 기록한다.

### 7.3 라우팅과 RAG

| 지표 | 계산식 | 목표 |
| --- | --- | ---: |
| Routing macro F1 | low/medium/manual-high fixture별 F1 평균 | >= 0.95 |
| Retrieval Recall@3 | 정답 정책이 top-3에 포함된 query 비율 | >= 0.85 |
| Retrieval MRR@3 | 첫 정답 정책 순위 reciprocal rank 평균 | >= 0.75 |
| Citation validity | citation이 실제 retrieved source와 일치하는 비율 | >= 0.98 |
| Unrelated retrieval rate | negative fixture에서 정책을 반환한 비율 | <= 0.05 |

### 7.4 리뷰 finding 품질

| 지표 | 계산식 | 목표 |
| --- | --- | ---: |
| Finding Precision@5 | 상위 5개 중 정답 defect/comment와 일치한 비율 | >= 0.70 |
| Finding Recall@5 | 정답 defect/comment 중 상위 5개가 포착한 비율 | >= 0.60 |
| Severe false-positive rate | 수정 후 승인 snapshot에서 근거 없는 high finding 비율 | <= 0.10 |
| Fix sensitivity | 수정 전 적중 finding이 수정 후 사라진 비율 | >= 0.80 |
| File reference validity | finding file이 PR changed files에 존재하는 비율 | 1.00 |
| Line reference validity | line이 제공 patch 범위에 존재하는 비율 | >= 0.98 |
| Secret masking pass rate | secret fixture가 prompt/trace/comment에 노출되지 않은 비율 | 1.00 |

`Finding Precision/Recall`의 일치는 file, category, 핵심 issue label 기준으로 판정한다. 문장
유사도만으로 정답 처리하지 않는다.

### 7.5 비용 효율

| 지표 | 계산식 | 목표 |
| --- | --- | ---: |
| 100 changed lines당 input token | input token / changed lines * 100 | baseline 대비 25% 감소 |
| 자동 리뷰당 비용 | route별 input/output token 기반 USD | route별 추세 공개 |
| high-only 대비 절감률 | 동일 평가 세트를 전부 high로 실행한 비용과 비교 | >= 25% |

품질 KPI를 충족하지 못한 비용 절감은 성공으로 계산하지 않는다.

## 8. 성능 개선 내용과 남은 작업

### 이번 반영

1. CI를 lint/smoke와 test 병렬 job으로 분리하고 pip cache를 사용한다.
2. low/medium/high별 changed file 수와 patch 문자 예산을 둔다.
3. 최대 finding 수와 최대 output token을 route별로 제한한다.
4. RAG를 top-3로 제한하고 overlap이 없는 정책 fallback을 제거한다.
5. 배포 후 policy index를 자동 동기화한다.

### 추가 구현 필요

1. 각 단계 timestamp를 영속화해 latency breakdown을 자동 집계한다.
2. Upstage/GitHub timeout과 rate limit에 exponential backoff와 최대 재시도를 적용한다.
3. PR diff가 큰 경우 위험 파일, 실행 코드, test 변경 우선순위로 context를 선택한다.
4. GitHub installation token을 만료 시각까지 cache해 반복 발급을 줄인다.
5. repository별 정책 sync와 embedding 기반 hybrid retrieval을 구현한다.

## 9. 발표에서 사용할 주장

다음처럼 측정 결과를 붙여 말할 수 있어야 한다.

> CI 이후 자동 리뷰를 diff 크기별로 측정했고 small PR p95는 N초, medium PR p95는 N초였다.
> 정책 검색은 30개 label query에서 Recall@3 N%, citation validity N%를 기록했다. 대형
> 오픈소스 PR의 changes-requested snapshot에서 maintainer finding Recall@5 N%, 수정 후 승인
> snapshot의 severe false-positive rate N%를 기록했다. 모든 PR을 high로 처리하는 기준선보다
> 비용은 N% 낮았고 동일한 품질 하한을 충족했다.

측정 전에는 숫자를 채우지 않고 `baseline pending`으로 둔다. 목표와 실측을 같은 표에서
구분해 과장된 100% 성공 또는 근거 없는 5분 이내 주장을 피한다.
