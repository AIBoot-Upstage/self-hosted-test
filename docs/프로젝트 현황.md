# AI Code Review Agent 프로젝트 현황

> 이 문서는 코드를 직접 읽지 않고도 프로젝트의 현재 상태를 파악하기 위한 **단일 기준 문서**다.
> 기능, 구조, 배포, 검증 결과가 바뀌면 코드 변경과 같은 commit에서 이 문서를 갱신한다.

- 마지막 갱신일: 2026-07-13
- 현재 단계: GCP VM 배포 및 실제 GitHub App PR 리뷰 동작 확인 후 품질·성능 고도화 단계
- 프로젝트 버전: `0.1.0`
- 운영 모델: `solar-pro3`
- 자동 리뷰 방식: CI 이후 실패 원인 우선 또는 정책 기반 표준 리뷰
- 선택 리뷰 방식: 사용자가 GitHub Check 버튼으로 심층 리뷰 요청
- 저장소 정책 표준 경로: 기본 브랜치의 `.github/ai-review-policy.md` (수집 기능은 미구현)
- 최근 로컬 검증: 단위 테스트 50개, Ruff, 정책 하네스 fixture benchmark와 실제 Solar3 PR diff 통과

## 1. 한눈에 보기

AI Code Review Agent는 GitHub Pull Request의 diff와 CI 결과를 분석하고, 상황에 맞게 같은
Upstage Solar3 모델의 추론 강도를 조절하는 GitHub App이다. CI가 실패하면 원인 해결을
우선하고, 통과하면 관련 내부 정책을 검색해 리뷰한다. 아키텍처, 복잡도, 코드 간소화처럼 더
넓은 관점은 사용자가 필요할 때만 심층 리뷰로 실행한다.

현재는 GitHub App webhook 수신부터 CI 결과 수집, LangGraph workflow, LiteLLM을 통한 실제
Solar3 호출, Postgres 저장, Langfuse 관측, GitHub Check와 댓글 게시까지 end-to-end 동작한다.
GCP VM에는 GitHub Actions CD로 Docker image가 배포되며 Caddy가 HTTPS를 제공한다.

## 2. 해결하려는 문제

| 문제 | 이 프로젝트의 접근 |
| --- | --- |
| CI 실패와 설계 이슈를 같은 깊이로 리뷰 | CI 결과에 따라 실패 리뷰와 표준 리뷰를 분리 |
| 팀 정책이 리뷰어마다 다르게 적용 | diff 신호로 관련 skill과 정책 유형을 고르고 근거 chunk만 사용 |
| 모든 PR에 고비용 추론 사용 | 자동 리뷰는 low/medium, high는 사용자 선택으로 제한 |
| 큰 diff가 latency와 token을 증가 | route별 파일 수, patch 길이, output token 예산 적용 |
| 댓글 생성만으로 품질을 판단 | Langfuse, Postgres와 정량 KPI로 시간·품질·비용을 측정 |

## 3. 현재 사용자 경험

1. 개발자가 GitHub App이 설치된 저장소에 PR을 생성하거나 새 commit을 push한다.
2. GitHub Actions의 `lint_smoke`와 `test` job이 병렬로 실행된다.
3. 두 결과를 합치는 `CI / ci`가 완료된다.
4. GitHub의 `check_suite.completed` webhook이 서비스로 전달된다.
5. 서비스가 PR, changed files, check runs를 GitHub API로 조회한다.
6. `AI Code Review` pending Check를 생성하고 리뷰를 실행한다.
7. 결과를 Postgres에 저장하고 GitHub 댓글과 Check 결과를 완료한다.
8. 사용자는 필요하면 Check의 `심층 리뷰 실행` 버튼으로 두 번째 리뷰를 요청한다.

PR 요약 댓글은 내부 추론 등급이나 추상적인 위험도를 전면에 표시하지 않는다. 사용자에게는
`변경 요약`, `변경 파일별 변경 요약` 표, `리뷰` 순서로 실제 변경과 조치할 내용을 보여준다.
finding의 severity는 정렬과 Check 결론을 위한 내부 값으로 유지하되 댓글 제목에서는 숨긴다.

```text
PR opened/synchronize
  -> CI: lint_smoke + test 병렬
  -> CI / ci 집계
  -> check_suite.completed webhook
  -> GitHub App installation token 발급
  -> PR diff/check 결과 조회
  -> AI Code Review pending Check
  -> LangGraph review workflow
  -> 신뢰된 script로 diff 신호 추출, skill·정책 선택
  -> Postgres 저장
  -> GitHub comment + Check 완료
  -> 선택적으로 심층 리뷰 실행
```

Branch protection을 적용하는 저장소에서는 `CI / ci`와 `AI Code Review`를 required check로
사용한다. 현재 프로젝트 저장소에 GitHub App을 반드시 설치해야 하는 것은 아니며, 실제 서비스
검증용 대상 저장소에 설치해 사용할 수 있다.

현재 GitHub App은 소유 Organization 내부 설치와 운영 검증까지 완료된 상태다. 다른 개인이나
Organization이 같은 App을 설치하는 공개 설치, 설치 후 온보딩, 고객별 데이터 격리는 아직
구현·검증되지 않았다.

## 4. 리뷰 라우팅

`low`, `medium`, `high`는 서로 다른 모델이 아니라 `solar-pro3`에 전달하는
`reasoning_effort` 값이다. 사용자에게는 내부 등급보다 리뷰 목적을 표시한다.

| 사용자 표시 | 내부 route | 조건 | RAG | 추론 강도 | 역할 |
| --- | --- | --- | --- | --- | --- |
| 실패 원인 우선 리뷰 | `simple_failure_review` | syntax, lint, test 중 하나 실패 | 사용 안 함 | low | 실패 근거, 원인과 수정 순서 |
| 정책 기반 표준 리뷰 | `policy_context_review` | 실패 check 없음 | 정책이 있으면 사용 | medium | 정책, 테스트, 스타일, API 계약 |
| 선택형 심층 리뷰 | `deep_quality_review` | 사용자가 Check action 클릭 | 정책이 있으면 사용 | high | 구조, 보안, 시간·공간 복잡도, 간소화 |

고위험 경로나 600줄 초과 diff, 20개 초과 파일은 자동 high로 보내지 않는다. 표준 리뷰의
판단 근거에 심층 리뷰 권고를 추가하고 최종 실행은 사용자에게 맡긴다.

## 5. LangGraph 실행 구조

배포 환경은 실제 LangGraph `StateGraph`를 사용한다. 로컬에서 LangGraph 의존성이 없을 때는
동일한 node와 edge 순서를 따르는 fallback executor가 테스트 편의를 위해 동작한다.

```text
create_review
  -> extract_features
  -> select_route
  -> select_harness
  -> retrieve_policies | skip_policy_retrieval
  -> build_prompt
  -> call_llm
  -> validate_findings
  -> assemble_result
  -> persist_result
  -> publish_comment
  -> complete_review
```

현재 node 상태는 한 실행 안에서만 전달된다. LangGraph checkpoint 저장소와 중단 지점부터의
재개는 구현되어 있지 않다. API의 일반 리뷰 요청과 GitHub webhook 처리는 FastAPI background
task에서 실행되며 Redis, Celery 같은 durable queue는 없다.

`validate_findings`는 모델 응답을 게시하기 전에 changed file, right-side diff line, 정책 출처,
선택 지식 카드 ID, 카드별 severity 상한, 중복과 route별 최대 finding 수를 검증한다. 선택 카드가
있는데 `knowledge_card_id`가 없거나 허용 목록 밖이면 해당 finding을 폐기한다. 검증된 line finding은
GitHub pull request review의 inline comment로 게시하고, line을 검증하지 못한 항목과 전체 요약은
PR-level comment로 보낸다.
inline API가 실패하면 모든 finding을 기존 comment 형식으로 fallback한다.
품질 gate의 수신·승인·제거 사유별 건수는 review 결과와 Postgres JSON payload에 저장한다.

`select_harness`는 서비스 image가 소유한 deterministic script로 CI·diff 신호를 추출하고 관련
`SKILL.md`, 근거·오탐 방지 지식 카드와 정책 유형을 고른다. `build_prompt`는 작은 PR에는 하나의 prompt를 만들고 큰 PR에는
route별 전체 상한 안에서 파일 최대 4개와 patch 6,000~7,000자 단위의 batch를 만든다. 각
batch에는 선택된 skill과 최대 2개 정책 chunk만 들어간다. `call_llm`은 최대 4개 batch를 병렬로
호출하고 finding, token usage, 변경 요약과 파일별 요약을 합친 뒤 기존 품질 gate로 전달한다.
파일별 요약은 실제 batch의 changed file 경로만 허용하며 모델이 누락하거나 존재하지 않는 경로를
반환하면 서버가 입력 파일 순서와 변경 줄 수를 기준으로 보완한다.

## 6. 정책 하네스와 RAG

### 하네스 실행 계약

```text
정책 Markdown -> heading chunk/index
PR diff·CI -> 신뢰된 diff_signals script -> 관련 skill·knowledge card 선택
skill의 policy_types -> 후보 정책 제한 -> batch별 top-2 검색
선택 skill + knowledge card + 정책 근거 + diff -> Solar3 review
```

- `review_harness/manifest.json`이 route별 최대 skill 수와 policy 유형 연결을 정의한다.
- `review_harness/scripts/diff_signals.py`는 API, 보안, 테스트, 성능, 신뢰성, 입력 경계, 데이터
  무결성, 의존성·workflow, frontend, 문서 계약 신호를 추출한다. diff header와 삭제 line은
  선택 근거에서 제외하며, 입력 경계는 관련 patch marker가 있을 때만 활성화한다.
- `review_harness/skills/*/SKILL.md`는 해당 관점에서 근거를 찾는 절차이며 선택된 것만 주입한다.
- `review_harness/references/knowledge-cards.json`은 구체적 검사 항목, 필요한 증거, 오탐 방지 조건과
  severity 상한을 정의한다. 선택된 skill 안에서 signal과 path/patch marker가 맞는 카드만 주입하며,
  일반 API 경로만으로 입력 sink나 breaking change 카드를 선택하지 않는다.
- 범용 동작·실패 경로 카드는 사용자 선택형 심층 리뷰에서만 사용한다. 자동 표준 리뷰는 구체적인
  API·보안·테스트·성능 등 카드가 선택될 때만 finding을 허용해 precision을 우선한다.
- `review_harness/references/sources.json`은 지식 카드와 skill의 외부 근거 ID를 검증한다. URL과
  외부 원문은 운영 prompt에 반복 주입하지 않는다.
- 저장소에서 가져온 임의 script나 skill은 실행하지 않는다. 서비스가 배포한 읽기 전용 자산만
  실행하는 것이 현재 보안 경계다.
- 정책 후보는 PR 단위로 찾고 실제 prompt에는 batch와 관련된 최대 2개 chunk만 포함한다.
- prompt에는 선택 skill과 카드 지침을 사용하지만 DB 결과에는 지침 원문을 제외하고 ID, 제목,
  근거 source ID, 정책 유형과 점수만 저장한다.

### 현재 구현

- `policies/`의 공통 정책 Markdown 6개를 사용한다.
- heading 단위, 최대 1,800자로 chunk를 나눈다.
- 배포 후 `policy_chunks` 테이블에 정책을 다시 동기화한다.
- PR 제목, 파일 경로, check summary, patch의 token overlap으로 관련도를 계산한다.
- 하네스가 선택한 정책 유형 안에서 batch별 overlap 상위 2개만 prompt에 전달한다.
- finding은 검색된 정책의 source를 `policy_source`로 표시할 수 있다.

현재 공통 정책은 다음과 같다.

| 정책 | 범위 |
| --- | --- |
| `api-contract.md` | API 계약과 호환성 |
| `github-review-workflow.md` | GitHub App과 리뷰 실행 흐름 |
| `testing-and-routing.md` | 테스트, CI, route 규칙 |
| `security-and-privacy.md` | secret, 인증, 개인정보 |
| `performance-and-maintainability.md` | 복잡도, 성능, 유지보수성 |
| `observability-and-reliability.md` | 로그, trace, 오류와 신뢰성 |

### 아직 구현되지 않은 부분

- embedding 생성과 vector 또는 hybrid 검색
- 설치 대상 repository 기본 브랜치의 `.github/ai-review-policy.md` 자동 수집
- repository별 정책 격리와 source commit SHA 관리
- retrieval fixture에 대한 Recall@2, MRR@2 자동 평가

### 확정된 저장소 정책 계약

- 저장소 정책의 표준 경로는 `.github/ai-review-policy.md`다.
- 리뷰에는 PR head가 아니라 base/default branch에서 읽은 정책만 사용한다. 같은 PR에서 정책을
  약화한 뒤 그 변경을 기준으로 리뷰를 우회하지 못하게 하기 위한 보안 경계다.
- 정책 원문은 installation ID, repository ID, source commit SHA와 함께 격리·버전 관리하는 것을
  목표로 한다.
- 초기 공개 버전은 저장소 정책을 우선한다. 여러 저장소가 공유하는 Organization 정책과 웹
  업로드는 멀티테넌트 저장 구조와 사용자 인증을 갖춘 뒤 추가한다.
- 현재 배포된 `policies/*.md`는 서비스 공통 정책이며 위 저장소 정책을 아직 자동으로 읽지 않는다.

Postgres schema에는 `VECTOR(1536)` 컬럼이 있지만 현재 값과 검색에는 사용하지 않는다. 따라서
현재 기능을 “pgvector semantic search”라고 설명하지 않고, “Postgres-backed lexical policy
retrieval을 사용하는 정책 하네스”라고 설명한다.

## 7. LLM과 관측

### LLM 호출

- LiteLLM은 별도 proxy나 dashboard가 아니라 API 컨테이너 안의 Python SDK다.
- OpenAI 호환 endpoint인 `https://api.upstage.ai/v1`로 `solar-pro3`를 호출한다.
- route에 따라 `reasoning_effort`와 prompt/file/token 예산을 다르게 전달한다.
- 생성 상한은 빠른/표준/심층 리뷰별 `4,096/8,192/16,384`다. Solar3 reasoning도 생성 예산을
  사용할 수 있어 과거 4,000 단일 상한보다 표준·심층 경로에 여유를 둔다.
- `LLM_MODE=mock`에서는 외부 API 없이 deterministic 결과로 로컬·CI smoke test를 수행한다.

### 관측 위치

| 확인 대상 | 위치 |
| --- | --- |
| 실제 prompt, 응답, token, LLM latency, route metadata | Langfuse Cloud |
| review 결과, route, model, usage payload | Postgres `review_runs` |
| 정책 원문과 chunk | Postgres `policy_chunks`와 `policies/` |
| webhook과 실행 오류 | API container logs |
| CI와 배포 단계 | GitHub Actions |
| 최종 사용자 결과 | GitHub PR Check와 comment |

Langfuse는 LiteLLM callback으로 연결된다. 비용은 provider/model 가격 metadata가 인식되는 경우에만
계산되므로 `cost_usd`가 비어 있어도 실제 API가 호출되지 않았다는 뜻은 아니다. 실제 호출 여부는
provider, model, token usage, Upstage request metadata와 응답을 함께 확인한다.

## 8. 리뷰 성능 개선 상태

### 반영된 최적화

| 영역 | 반영 내용 | 직접 줄이는 비용 |
| --- | --- | --- |
| CI | lint/smoke와 test 병렬 실행, pip cache | `after_checks` 모드의 CI 대기 시간 |
| 라우팅 | 실패 리뷰 low, 표준 리뷰 medium, high는 사용자 선택 | 불필요한 고강도 추론 |
| 입력 prompt | route별 최대 파일 8/20/30개, patch 12k/30k/50k 문자 | input token과 LLM 처리 시간 |
| 큰 PR 분할 | batch당 파일 최대 4개, patch 6k~7k, 동시 호출 최대 4개 | 추론이 completion 상한을 모두 소진하는 문제 |
| 출력 prompt | route 전체 finding 3/6/8개를 batch 수로 나눈 상한 | batch마다 전체 quota를 반복하는 응답 방지 |
| LLM 호출 | route별 `max_tokens` 4,096/8,192/16,384 | 4,000 조기 종료 방지와 비용 상한 |
| 정책 하네스 | script 신호로 관련 skill·정책 유형 선택 | 모든 정책·절차의 반복 주입 방지 |
| 지식 카드 | skill 안에서 근거·오탐 조건을 path/patch별 선택 | 무관한 검사와 추상적 지적 감소 |
| 정책 검색 | batch별 최대 2개, overlap 없는 fallback 제거 | 무관 context와 prompt token |
| 배포 | 배포 후 정책 index 자동 sync | 오래된 정책으로 인한 재작업 위험 |

PR #11의 동일 head SHA와 Solar3 medium 설정에서 하네스 정제 전후를 단회 비교했다. 70개 changed
file 중 route 예산으로 선택된 파일을 7개 batch로 처리한 결과, 범용 카드와 batch별 전체 quota를
사용한 첫 실행은 prompt/completion `27,607/22,562` token, LLM `64,593ms`, raw finding 22개였다.
batch별 quota 분배 후에는 `28,062/19,195` token, `46,132ms`, 5개였고, 범용 카드를 표준 route에서
제외한 실행은 `27,324/9,782` token, `32,730ms`, 1개였다. 첫 실행 대비 마지막 전체 실행의
completion은 56.6%, LLM 시간은 49.3% 감소했다. 남은 외부 LLM 호출 오탐의 guard를 보강한 해당
batch 재실행은 raw finding 0개였다.

같은 PR과 설정을 각 단계에서 한 번씩 실행한 개발 측정이므로 일반화된 성능 KPI나 최종 품질
증명은 아니다. 전체 최종 하네스의 반복 실행과 여러 실제 PR 표본으로 p50/p95, finding precision을
다시 측정해야 한다.

### 아직 남은 병목

- PR 생성부터 리뷰까지의 체감 시간에는 GitHub Actions queue와 전체 CI 시간이 포함된다.
- GitHub API의 PR, files, checks 조회와 installation token 발급이 순차 실행되고 cache가 없다.
- LLM 호출 timeout은 90초이며 timeout/rate-limit retry와 backoff가 없다.
- 단계별로는 LLM latency만 저장하며 `ci_wait`, GitHub fetch, RAG, publish 시간은 영속화하지 않는다.
- Postgres RAG는 매 리뷰마다 모든 policy chunk를 읽어 Python에서 점수를 계산한다.
- idempotency 확인 시 기존 리뷰 목록을 읽어 set으로 만들기 때문에 이력이 커지면 비용이 증가한다.
- FastAPI background task와 단일 API process 구조라 동시 PR 처리와 장애 복구에 한계가 있다.
- PR #11의 GitHub Check는 이번 branch가 배포되기 전 서버 image에서 실행돼 구버전 단일 LLM 호출의
  90초 timeout으로 실패했다. 새 batch 하네스의 실제 PR diff 실행 성공과 구분한다.

다음 성능 작업은 추측성 최적화보다 단계 timestamp 수집을 먼저 추가하고, 실제 trace에서 가장 큰
구간부터 개선해야 한다. PR 생성 시각부터의 `ci_wait_ms`와 CI 완료 webhook 이후의
`agent_latency_ms`를 반드시 분리한다.

## 9. RAG 성숙도 판정

### 결론

현재 구현은 외부 정책을 검색하고 그 결과를 LLM prompt에 추가해 답변을 생성하므로 최소 정의상
**RAG라고 부를 수 있다**. 다만 정확한 표현은 **서비스 공통 정책 기반 lexical RAG MVP**다.
embedding, vector search, reranker, 저장소별 지식 동기화와 검색 품질 평가가 적용된 production
RAG로 설명하면 과장이다.

### RAG 구성요소 충족 여부

| 구성요소 | 현재 상태 | 판정 |
| --- | --- | --- |
| 외부 지식 원본 | `policies/*.md` 6개 | 충족 |
| chunk/index | heading 단위 chunk, Postgres `policy_chunks` | 충족 |
| query 구성 | PR 제목, 파일 경로, check 요약, patch 일부 | 충족 |
| retrieval | 하네스 유형 제한 후 batch별 token overlap 상위 2개 | 기초 수준 충족 |
| prompt augmentation | source, section, content, score를 prompt에 주입 | 충족 |
| grounded output | source membership 사후 검증 | 출처 존재는 강제, 내용의 의미적 지지는 미검증 |
| semantic retrieval | embedding/vector/hybrid search | 미구현 |
| repository knowledge | 설치 저장소별 정책 수집과 격리 | 미구현 |
| quality evaluation | 하네스 선택 fixture 11개 | 선택 단계 baseline만 측정 |

2026-07-13 로컬 확인에서는 6개 문서가 24개 chunk로 분리됐다. 최신 실제 Solar3 sample 호출에서는
`change-correctness`, `api-contract` skill과 `behavior-edge-and-failure-path` 카드가 선택됐고,
`api-contract.md`의 API-002/API-004 두 chunk가 prompt와 finding 출처에 반영됐다. 모델이 반환한
finding 2개는 모두 허용된 `knowledge_card_id`와 검색된 `policy_source` 검증을 통과했다. 입력 sink
근거가 없는 additive API diff에서 이전에 선택되던 `input-boundary-safety`와 breaking 카드는 선택
조건을 좁힌 뒤 제외됐다. 이 실행의 Solar3 사용량은 prompt 1,939 token, completion 1,948 token,
LLM latency 23,924ms였다. 단일 합성 sample의 관측값이며 대표 latency나 품질 향상 수치가 아니다.

### 현재 품질 한계

- 동의어와 의미 유사성을 찾지 못하고 공통 단어가 많은 문서가 높은 점수를 받을 수 있다.
- TF-IDF/BM25가 아니라 단순 overlap이어서 흔한 token의 중요도를 낮추지 못한다.
- 정책 존재 여부와 현재 PR에 관련 정책이 검색됐는지가 route에서 구분되지 않는다.
- prompt는 정책을 “repository policy”라고 부르지만 실제로는 서비스에 배포된 공통 정책이다.
- retrieved set 밖의 `policy_source`는 제거하지만 정책 내용이 finding을 실제로 지지하는지는 판정하지 않는다.
- 허용된 지식 카드 ID는 강제하지만 모델이 카드의 `evidence_required`를 의미적으로 충족했는지는
  아직 자동 판정하지 않는다. 합성 sample에서도 diff 밖의 예외 동작을 추정한 finding이 남았다.
- 긴 section은 문자 수로 잘리므로 문장이나 규칙 경계가 끊길 수 있다.
- Postgres와 pgvector를 사용하지만 `embedding` 컬럼은 비어 있고 SQL vector 검색은 수행하지 않는다.

RAG 고도화 순서는 30개 이상의 labeled query fixture와 Recall@2/MRR@2 baseline 생성, BM25 또는
가중 lexical 검색, repository scope와 source SHA 추가, citation 검증, embedding+lexical hybrid
검색 및 reranking 비교가 적절하다. vector 검색 도입 자체가 아니라 동일 fixture에서 품질이
실제로 향상되는지를 기준으로 선택한다.

### 하네스 선택 baseline

2026-07-13 서비스 소유 fixture 11개를 대상으로 `evaluate_harness`를 실행한 결과는 다음과 같다.
외부 검토 지식은 Google Engineering Practices·Testing Blog, OWASP, GitHub Docs, Google AIP·SRE,
PostgreSQL, W3C, OpenAI의 공식 자료 23개에서 정제한 26개 카드다. 모든 skill과 카드는 하나 이상의
source ID를 가져야 하며, 알 수 없는 source·중복 ID·HTTP URL·필수 근거 필드 누락은 하네스 로드
단계에서 거부한다.

해석하지 않는다. CI가 같은 fixture benchmark를 실행해 manifest·skill 변경 회귀를 차단한다.

### 정책 주입이 리뷰 품질에 미치는 영향

현재 확인된 것은 **정책 검색과 prompt 주입의 기능적 연결**이다. 관련 정책이 검색되고
`policy_source`와 함께 prompt 및 결과에 전달되며, 선택 지식 카드의 증거·오탐 지침과 허용 ID가
실제 Solar3 prompt에 포함된다. 응답의 card ID와 정책 source membership을 사후 검증하는 테스트도
통과한다. 하지만 이것만으로 정책이
finding의 정확도나 유용성을 높였다고 결론 내릴 수는 없다.

2026-07-13 기준으로 아직 수행하지 않은 검증은 다음과 같다.

- 동일한 실제 PR과 동일한 Solar3 설정에서 정책 미주입/주입 결과를 비교하는 A/B 또는 ablation
- 정책 위반 정답이 있는 fixture에서 Finding Precision@5와 Recall@5 비교
- 정상 변경에서 정책 주입 전후의 false-positive rate 비교
- 인용 정책과 지식 카드가 finding의 주장 내용을 실제로 지지하는지 의미 기반 판정
- 여러 회 반복 호출을 통한 비결정성 통제와 통계 비교

따라서 현재 발표 가능한 사실은 “정책을 리뷰 입력과 출처로 반영한다”까지이며, “정책으로 리뷰
품질이 향상됐다”는 주장은 위 실험의 결과가 나온 뒤에만 사용한다.

### 대규모 오픈소스 PR 데이터 적용 상태

CPython, Django, Kubernetes, Rust의 maintainer review를 사용하는 평가 방법과 KPI는 문서에
설계되어 있다. 공개 GitHub PR/review/comment 수집 CLI와 maintainer/bot 분류까지는 구현했지만,
commit snapshot 복원, maintainer finding 정답 label, 평가 runner와 결과 report는 아직 없다.
`sample-data/`에는 로컬 동작 확인용 PR payload 2개만 있다.

단순히 merged PR을 “좋은 코드”, closed 또는 반려 PR을 “나쁜 코드”로 label하지 않는다. 실제
적용 시에는 maintainer의 `CHANGES_REQUESTED` 시점 commit을 수정 전 snapshot으로, 같은 PR의
수정 후 `APPROVED` 또는 merge commit을 수정 후 snapshot으로 사용한다. duplicate, stale, 방향
변경, CI 장애 같은 비품질 사유로 닫힌 PR은 제외하고, 모델 입력에서는 maintainer comment와
이후 patch를 제거해 정답 유출을 방지한다.

초기 적용 단위는 다음과 같다.

1. 수정 전/후 PR 25쌍과 승인 상태 정상 PR 25개, 총 75 snapshot 수집
2. maintainer finding을 file, line, category, severity와 핵심 issue label로 정규화
3. 동일 snapshot에 정책 미주입과 정책 주입 리뷰를 각각 실행
4. Precision@5, Recall@5, severe false-positive rate, fix sensitivity 비교
5. 품질이 개선된 prompt와 정책만 운영 후보로 승격하고 Langfuse에 prompt version 기록

## 12. 배포 구조

### 현재 자동 배포 순서

1. `main` push로 CI가 실행된다.
2. `lint_smoke`, `test`, aggregate `ci`가 모두 성공한다.
3. `Deploy to GCP VM` workflow가 `workflow_run`으로 시작된다.
4. Actions runner에서 commit SHA tag의 Docker image를 build한다.
5. image를 `docker save | gzip`으로 archive하고 Compose, Caddyfile, policies를 묶는다.
6. GitHub Actions가 WIF로 GCP service account 권한을 얻는다.
7. IAP tunnel의 `gcloud compute scp`로 VM에 archive와 배포 script를 전송한다.
8. VM에서 `docker load` 후 `docker compose up -d --no-build`를 실행한다.
9. 공통 정책을 Postgres에 sync하고 `/healthz`를 확인한다.

현재 workflow는 GHCR이나 Docker Hub에서 image를 pull하지 않는다. image archive를 runner에서
VM으로 직접 전송한다. VM의 `~/ai-code-review-agent-deploy/.env`는 배포 artifact에 포함되지 않으며
서버가 보관한다.

### 배포 인증과 설정 소유권

| 위치 | 값의 종류 |
| --- | --- |
| GitHub Variables | GCP project, zone, instance, image name, deploy target |
| GitHub Secrets | WIF provider, deploy service account |
| VM `.env` | Upstage key, GitHub App 설정, webhook secret, Langfuse key, DB 비밀번호, 실행 mode |

GCP service account와 WIF를 사용하므로 장기 GCP JSON key는 GitHub에 저장하지 않는다. GitHub
App private key와 애플리케이션 runtime secret은 현재 CD가 전달하지 않으며 VM `.env`에서 읽는다.

### 현재 배포 한계

- 단일 API container를 Compose로 recreate하므로 완전한 무중단 rolling deployment가 아니다.
- 이전 image archive를 registry에 체계적으로 보관하지 않아 자동 rollback이 없다.
- health check 실패 시 이전 image로 되돌리는 단계가 없다.
- `docker image prune -f`로 미사용 image를 정리하므로 장기 rollback 보관소 역할을 하지 않는다.

무중단과 안전한 rollback이 필요하면 registry의 immutable SHA image, blue/green 또는 두 replica,
readiness 확인 후 traffic 전환, 실패 시 이전 tag 복구를 별도 구현해야 한다.

| `GITHUB_WEBHOOK_REVIEW_MODE` | `after_checks` | CI 완료 후 리뷰, 운영 기본 |
|  | `pull_request` | CI 없이 PR 이벤트로 즉시 리뷰 |
|  | `all` | 두 이벤트 모두 처리, 중복 주의 |
| `STORAGE_BACKEND` | `postgres` / `local` | DB 또는 JSON fallback |
| `RAG_BACKEND` | `postgres` / `local` | DB 정책 index 또는 파일 검색 |
| `CD_DEPLOY_TARGET` | `gcp-vm` | main CI 성공 후 VM 배포 |
|  | `local-only` | image build만 검증 |

전체 환경변수와 값의 출처는 `.env.example`을 기준으로 한다. 실제 secret 값은 문서나 Git에
기록하지 않는다.

## 15. 알려진 한계와 위험

| 우선순위 | 항목 | 현재 영향 |
| --- | --- | --- |
| 높음 | durable queue와 workflow checkpoint 없음 | process 종료 시 진행 중 리뷰 복구 불가 |
| 높음 | 외부 API retry/backoff와 자동 복구 미완성 | timeout, rate limit이 리뷰 실패로 이어질 수 있음 |
| 높음 | latency·품질 baseline 미측정 | 개선 효과를 수치로 아직 증명하지 못함 |
| 높음 | 공개 설치 온보딩과 테넌트 격리 미구현 | 외부 고객 저장소를 안전하게 서비스할 수 없음 |
| 높음 | 일반 저장소의 CI 완료 판정 미구현 | 여러 check suite 중 하나가 먼저 끝나면 리뷰가 너무 일찍 시작될 수 있음 |
| 중간 | 공통 lexical RAG | 저장소별 문맥과 의미 유사 표현 검색이 제한됨 |
| 중간 | automatic rollback과 무중단 배포 없음 | 배포 중 짧은 중단 및 실패 복구 수동 처리 가능성 |
| 중간 | SSE가 in-memory | 재시작 또는 다중 replica에서 event 연속성 없음 |
| 중간 | installation token cache 없음 | 동일 실행 중 GitHub token 발급 호출이 반복될 수 있음 |
| 낮음 | 운영 UI 없음 | 운영자는 GitHub, Langfuse, DB, container logs를 나눠 확인 |

### Production 준비도 감사

현재 end-to-end 데모와 단일 Organization 운영 검증에는 사용할 수 있지만 공개 다중 고객
production 서비스로 부를 수는 없다. 점검에서 확인한 임시·MVP 경계는 다음과 같다.

| 항목 | 판정 | 처리 상태 |
| --- | --- | --- |
| Postgres host 포트 | 운영 설정 불일치 | 외부 전체 bind를 제거하고 `127.0.0.1`로 제한 완료 |
| 리뷰 작업 실행 | MVP | FastAPI background task라 재시작 시 복구되지 않음 |
| LLM/GitHub 장애 | MVP | timeout은 있으나 retry/backoff와 재실행 queue 없음 |
| 멱등성 | 부분 구현 | 완료 이력 조회 기반이며 DB unique lock이 없어 동시 webhook race 가능 |
| 정책 데이터 | MVP | 공통 정책만 사용하고 repository/installation 격리 미구현 |
| 정책 동기화 | MVP | 전체 `policy_chunks` 삭제 후 재삽입이라 tenant별 증분 sync가 아님 |
| LangGraph 상태 | MVP | durable checkpoint와 중단 지점 재개 없음 |
| 배포 | 단일 서버 검증용 | recreate 배포, registry rollback과 무중단 전환 없음 |
| secret 관리 | 단일 VM 운영용 | VM `.env` 사용, Secret Manager runtime 주입 미구현 |
| 하네스 script | 안전 경계 적용 | 서비스 소유 script만 실행하며 사용자 제공 script 실행은 금지 |

발표에서는 “배포·실사용 흐름이 완성된 MVP”라고 표현하고 “production-ready” 또는 “품질 향상
검증 완료”라고 표현하지 않는다.

## 16. 다음 우선순위

1. `installation_id`와 `repository_id` 기준의 고객·정책·리뷰 데이터 격리 schema를 만든다.
2. 기본 브랜치의 `.github/ai-review-policy.md` 수집, source SHA, 갱신과 삭제 흐름을 구현한다.
3. 설치·저장소 추가·제거·App 삭제 webhook과 공개 설치 온보딩 상태를 처리한다.
4. 저장소마다 다른 CI를 위해 대상 check 설정, 전체 완료 판정, 안정화 시간과 최대 대기시간을 구현한다.
5. webhook 수신부터 GitHub 게시까지 단계 timestamp를 저장하고 diff/route별 p50·p95를 측정한다.
6. 현재 11개 하네스 선택 fixture를 36개로 확장하고 라우팅·정책 검색 baseline을 만든다.
7. GitHub와 Upstage 호출에 제한된 exponential backoff, 실패 상태, 재실행 정책을 추가한다.
8. 같은 repository, PR, head SHA, mode의 동시 실행을 DB 수준에서 방지한다.
9. 공개 PR 수집기로 CPython, Django, Kubernetes, Rust의 maintainer review snapshot을 확장한다.
10. immutable image registry와 rollback 또는 blue/green 배포를 설계한다.

### 공개 GitHub App 목표 구조

운영 Organization이 하나의 GitHub App과 private key를 소유하고, 외부 사용자는 별도 App을 만들지
않고 그 App을 자신의 개인 계정 또는 Organization에 설치한다. 설치마다 발급되는
`installation_id`를 고객 경계로 사용하고, repository 선택과 정책·리뷰 데이터는 repository ID로
추가 격리한다. 각 GitHub API 호출에는 해당 installation의 단기 access token만 사용한다.

첫 공개 beta는 Marketplace 등록 없이 공개 App의 설치 링크를 제한적으로 공유한다. 다음 조건이
갖춰진 뒤 GitHub App 설정을 public으로 전환한다.

1. 설치와 저장소 변경 webhook을 멱등하게 처리하고 uninstall 시 token, cache, 고객 데이터 정리
2. 저장소 정책과 리뷰 이력이 다른 installation에서 조회되지 않는 격리 테스트
3. 저장소마다 다른 CI check의 전체 완료 판정, 사용량 제한, 동시 실행 제어
4. 개인정보 처리방침, 데이터 보존·삭제 기준, 이용약관과 지원 연락처
5. 외부 API 장애 retry와 실패 상태 노출, 공개 HTTPS webhook, 운영 모니터링, backup과 rollback

웹 dashboard 없이 시작하는 beta에서는 설치 후 사용자가 선택한 저장소의 기본 브랜치에
`.github/ai-review-policy.md`를 commit하면 된다. 설치 계정 확인, 정책 업로드, 사용량 조회 같은
dashboard를 제공할 때 GitHub 사용자 OAuth callback, session, CSRF `state`와 추가 계정 schema를
도입한다. App 설치 권한과 웹 사용자 인증은 서로 다른 개념이다.
